# Comprehensive Guide to Docker & Kubernetes

This guide covers the core concepts, architecture, and practical implementations of Docker and Kubernetes.

---

## Part 1: Docker

Docker is a platform for developing, shipping, and running applications in isolated environments called **containers**. It solves the "it works on my machine" problem by packaging an application along with all of its dependencies into a single box.

### Core Concepts
1. **Image**: A read-only template with instructions for creating a Docker container. It includes the application code, libraries, tools, and dependencies.
2. **Container**: A runnable instance of an image. It is isolated from the host machine and other containers, operating almost like a lightweight virtual machine.
3. **Dockerfile**: A text document that contains all the commands a user could call on the command line to assemble an image.
4. **Volume**: A mechanism for persisting data generated by and used by Docker containers. Without volumes, data inside a container is lost when the container is deleted.
5. **Docker Compose**: A tool for defining and running multi-container Docker applications using a declarative YAML file.

### Common Docker Commands

```bash
# ==================== IMAGES ====================
docker pull <image_name>         # Download an image from Docker Hub
docker images                    # List all local images
docker rmi <image_id>            # Remove/Delete an image
docker build -t <name>:<tag> .   # Build an image from a Dockerfile in current dir

# ================== CONTAINERS ==================
docker run -d -p 8080:80 <image> # Run container detached (-d), mapping host:container ports
docker ps                        # List currently running containers
docker ps -a                     # List all containers (including stopped ones)
docker stop <container_id>       # Gracefully stop a running container
docker rm <container_id>         # Remove a stopped container
docker logs <container_id>       # View logs outputted by a container
docker exec -it <container_id> sh # Open an interactive shell inside a running container
```

### Dockerfile Examples

#### 1. Basic Spring Boot Java Dockerfile

```dockerfile
# Use an official OpenJDK runtime as a parent image
FROM eclipse-temurin:17-jre-alpine

# Set the working directory inside the container
WORKDIR /app

# Copy the packaged jar file from your host machine into the container
COPY target/my-spring-boot-app.jar app.jar

# Document the port that the application runs on
EXPOSE 8080

# The command that runs when the container starts
ENTRYPOINT ["java", "-jar", "app.jar"]
```

#### 2. Multi-Stage Build (Node.js/React App)
Multi-stage builds help keep the final image size small by leaving build dependencies behind and only keeping the final compiled artifacts.

```dockerfile
# --------- Stage 1: Build the application ---------
FROM node:18-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

# --------- Stage 2: Serve the application ---------
FROM nginx:alpine
# Copy the built assets from the previous 'build' stage
COPY --from=build /app/build /usr/share/nginx/html
EXPOSE 80
# Run Nginx in the foreground
CMD ["nginx", "-g", "daemon off;"]
```

### Docker Compose Example
Useful for spinning up interconnected services (e.g., your backend application and its PostgreSQL database at the same time).

`docker-compose.yml`:
```yaml
version: '3.8' # Docker compose version

services:
  backend-api:
    build: . # Builds from the Dockerfile in the current directory
    ports:
      - "8080:8080"
    environment:
      - SPRING_DATASOURCE_URL=jdbc:postgresql://postgres-db:5432/mydb
      - SPRING_DATASOURCE_USERNAME=myuser
      - SPRING_DATASOURCE_PASSWORD=mypassword
    depends_on:
      - postgres-db # Ensures DB starts before the API
    networks:
      - app-network

  postgres-db:
    image: postgres:14-alpine
    environment:
      POSTGRES_DB: mydb
      POSTGRES_USER: myuser
      POSTGRES_PASSWORD: mypassword
    ports:
      - "5432:5432"
    volumes:
      - pg-data:/var/lib/postgresql/data # Persist data across resets
    networks:
      - app-network

volumes:
  pg-data: # Declaring the named volume

networks:
  app-network:
    driver: bridge
```
*Run this stack using:* `docker-compose up -d`  
*Stop this stack using:* `docker-compose down`

## Part 2: Kubernetes (K8s)

Kubernetes is an open-source container orchestration platform that automates the deployment, scaling, and management of containerized applications. While Docker runs containers, Kubernetes manages thousands of running containers across multiple servers.

### Kubernetes Architecture
1. **Master Node (Control Plane)**: Manages the cluster.
   - **kube-apiserver**: The frontend of the K8s control plane (receives `kubectl` commands).
   - **etcd**: Consistent and highly-available key-value store used for all cluster background data.
   - **kube-scheduler**: Watches for newly created Pods and assigns them to worker nodes.
   - **kube-controller-manager**: Runs controller processes that handle node failures, replication numbers, etc.
2. **Worker Node**: The servers where workloads (containers) actually run.
   - **kubelet**: An agent running on each node that ensures containers are running cleanly inside a Pod.
   - **kube-proxy**: Maintains network routing rules on nodes.
   - **Container Runtime**: The software responsible for running containers (like `containerd` or `docker`).

### Core K8s Objects
- **Pod**: The smallest deployable unit in K8s. Usually encapsulates a single container (but can hold multiple tightly-coupled containers).
- **ReplicaSet**: Ensures a strictly configured number of pod replicas are running at any given time.
- **Deployment**: A higher-abstraction over ReplicaSets. Provides declarative updates, scaling, and rolling rollbacks for Pods.
- **Service**: An abstract way to expose an application running on multiple transient Pods as a single, static network service (IP/Port).
- **ConfigMap / Secret**: Mechanisms to inject environment variables, configs, or passwords into Pods without hardcoding them into the Docker image.
- **Ingress**: Manages routing of external HTTP/HTTPS traffic from outside the cluster to internal services.

### Common `kubectl` Commands

```bash
kubectl cluster-info                # View general cluster details
kubectl get nodes                   # List all worker/master nodes
kubectl get pods -n <namespace>     # List pods in a specific namespace
kubectl apply -f manifest.yaml      # Apply/Create resources from a K8s YAML file
kubectl delete -f manifest.yaml     # Delete resources from a YAML file
kubectl describe pod <pod_name>     # Show deep, detailed info/events about a pod
kubectl logs <pod_name>             # View logs of a specific pod
kubectl scale deployment <name> --replicas=5 # Scale a deployment to 5 pods
kubectl port-forward pod/<pod> 8080:80  # For testing, map local port to pod port
```

### Kubernetes Manifest Examples

In Kubernetes, we describe **desired state** using YAML files, and Kubernetes constantly works to match the actual state to the desired state.

#### 1. Deployment Manifest
This defines a desired state of 3 running instances (replicas) of an application.

`app-deployment.yaml`:
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app-deployment
  labels:
    app: backend
spec:
  replicas: 3
  selector:
    matchLabels:
      app: backend
  template: # --- The Pod Template (everything beneath this is a 'Pod') ---
    metadata:
      labels:
        app: backend # Must naturally match the selector above!
    spec:
      containers:
      - name: app-container
        image: my-dockerhub-user/my-app:1.0.0
        ports:
        - containerPort: 8080
        env: # Injecting Environment Variables
        - name: DATABASE_URL
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: db_url
        - name: DATABASE_PASSWORD
          valueFrom:
            secretKeyRef:
              name: app-secret
              key: db_password
        resources: # Limits and Requests (vital for K8s Scheduler)
          requests:
            memory: "256Mi"
            cpu: "250m" # 0.25 of a CPU core
          limits:
            memory: "512Mi"
            cpu: "500m"
```

#### 2. Service Manifest
Because Pods are ephemeral (they die and get new IP addresses constantly), we need a **Service** to act as a stable traffic cop in front of them.

`app-service.yaml`:
```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app-service
spec:
  type: LoadBalancer # Can be ClusterIP (internal only), NodePort, or LoadBalancer (provisions cloud LB)
  selector:
    app: backend # Routes traffic to any Pod with this label
  ports:
    - protocol: TCP
      port: 80       # Port exposed by the Service to the world
      targetPort: 8080 # Port where the container inside the Pod is actually listening
```

#### 3. ConfigMap & Secret Manifests
Used to keep environments identical but configuration dynamic.

`app-configmap.yaml`:
```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  db_url: "jdbc:postgresql://db-service:5432/mydb"
  environment: "production"
```

`app-secret.yaml`:
*Note: Secrets in pure K8s are base64 encoded, not encrypted by default.*
```yaml
apiVersion: v1
kind: Secret
metadata:
  name: app-secret
type: Opaque
data:
  # Base64 encoded values (e.g., generate via: echo -n 'supersecret' | base64)
  db_password: c3VwZXJzZWNyZXQ= 
```

#### 4. Persistent Volume Claim (PVC)
Used for databases or any stateful application where disk data needs to survive container crashes.

`db-storage.yaml`:
```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: postgres-pvc
spec:
  accessModes:
    - ReadWriteOnce # Can be mounted by a single node at one time
  resources:
    requests:
      storage: 5Gi # Requesting 5 GB of disk space
---
# Injecting the PVC into a Stateful Database Deployment
apiVersion: apps/v1
kind: Deployment
metadata:
  name: postgres
spec:
  replicas: 1
  selector:
    matchLabels:
      app: db
  template:
    metadata:
      labels:
        app: db
    spec:
      containers:
      - name: postgres
        image: postgres:14
        env:
          - name: POSTGRES_PASSWORD
            value: "password"
        volumeMounts:
        - mountPath: "/var/lib/postgresql/data"
          name: db-volume # References the volume declared below
      volumes:
      - name: db-volume
        persistentVolumeClaim:
          claimName: postgres-pvc # Connects to the PVC defined above
```

## 🚀 The End-To-End Developer Workflow

1. Write your Java/Node/Python code locally.
2. Write a `Dockerfile` that packages your application.
3. Test locally using `docker build` and `docker run` or `docker-compose`.
4. Run `docker push` to upload your container image to a container registry (Docker Hub, AWS ECR, GCP Artifact Registry).
5. Write Kubernetes K8s manifests (`Deployment`, `Service`, etc.) replacing the `image:` attribute with your newly pushed image.
6. Run `kubectl apply -f .` to apply your configurations to the cluster.
7. Kubernetes downloads the image, starts the Pods on different machines, connects them to the Service, and ensures your application runs robustly and scales seamlessly based on traffic.
